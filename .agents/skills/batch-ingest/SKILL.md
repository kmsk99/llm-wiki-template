# Batch Ingest: 대량 Raw → Wiki 일괄 승격

대량의 Markdown 파일(Obsidian vault 등)을 효율적으로 wiki/로 일괄 승격하는 파이프라인.
기존 `/project:ingest`가 1건씩 처리하는 반면, batch-ingest는 20-50건 단위 배치로 처리한다.

## 입력

$ARGUMENTS

- 인자가 없으면: raw/ 전체를 스캔하여 manifest에서 `미정`인 .md 파일을 모두 대상으로 한다.
- 디렉토리 경로가 주어지면: 해당 디렉토리 내 .md 파일만 대상으로 한다.
- `--mode <promote|merge|selective>`: 처리 모드 지정. 생략 시 Phase 2에서 파일별로 자동 분류.
- `--batch-size <N>`: 배치 크기 (기본값 30, 범위 20-50).
- `--dry-run`: 실제 파일 변경 없이 처리 계획만 출력한다.

## 처리 모드

### promote (빠른 승격)
이미 정제된 노트를 wiki/로 이동한다. 본문은 그대로 유지하고 frontmatter만 추가/보정한다.
- 대상: 구조화된 문장, 명확한 주제, 충분한 분량(200자 이상)의 노트.
- 처리: frontmatter 추가 → wiki/ 적절한 하위 디렉토리에 파일 생성 → manifest 갱신.

### merge (통합)
같은 주제의 파편화된 노트들을 하나의 wiki 문서로 통합한다.
- 대상: QMD 유사도 0.7 이상으로 클러스터링된 노트 그룹.
- 처리: 클러스터 내 노트들을 읽고 → 하나의 통합 문서 작성 → 모든 원본을 source에 기록 → manifest 갱신.

### selective (선별)
중요한 노트만 승격하고, 나머지는 raw/에 보존한다.
- 대상: 핵심 사실, 결정사항, 반복 절차가 포함된 노트.
- 처리: 중요도 평가 → 상위 노트만 승격 → 나머지는 manifest에 `보류` 표시.

## 실행 절차

### Phase 1: 대상 수집 및 Catalog 확인

1. 대상 파일 목록을 수집한다.
   - 인자가 없으면 raw/ 전체를 재귀 스캔하여 .md 파일을 수집한다 (.parsed.md 제외).
   - 디렉토리 경로가 주어지면 해당 디렉토리만 스캔한다.
2. `raw/.manifest.md`를 읽고, 미등록 파일이 있으면 manifest에 등록한다 (catalog 동작).
   - 유형, 날짜, 비고를 자동 추출한다.
   - 인제스트 상태는 `미정`으로 설정한다.
3. 이미 `완료` 상태인 파일은 대상에서 제외한다.
4. 총 대상 파일 수와 예상 배치 수를 출력한다.

### Phase 2: 클러스터링 및 모드 분류

5. QMD 시맨틱 검색으로 대상 파일들을 주제별로 클러스터링한다.
   - 각 파일의 제목과 첫 200자를 vec 검색하여 유사 파일을 그룹핑한다.
   - 유사도 0.7 이상인 파일들을 같은 클러스터로 묶는다.
6. 클러스터 결과를 바탕으로 wiki/ 폴더 구조를 제안한다.
   - 기존 wiki/ 디렉토리 구조(systems, processes, projects, decisions, playbooks, entities, glossary)에 매핑한다.
   - 기존 카테고리에 맞지 않으면 적절한 하위 디렉토리를 제안한다.
7. `--mode`가 지정되지 않았으면 각 파일/클러스터별로 모드를 자동 분류한다:
   - 단독으로 충분히 정제된 노트 → **promote**
   - 같은 주제의 2개 이상 파편 노트 → **merge** (클러스터 단위)
   - 메모 수준이거나 정보량이 적은 노트 → **selective** 후보
8. `--dry-run`이면 여기서 처리 계획을 출력하고 종료한다:
   - 클러스터 구조, 모드 분류 결과, 예상 wiki/ 경로, 배치 분할 계획.

### Phase 3: 배치 처리

9. 대상 파일을 배치 크기(기본 30)로 분할한다.
10. 각 배치를 순차 처리한다:

#### promote 처리
   a. 원본 내용을 읽는다.
   b. frontmatter를 추가/보정한다:
      - `title:` — 파일명 또는 첫 번째 제목에서 추출
      - `owner: knowledge-system`
      - `status: draft`
      - `updated:` — 오늘 날짜
      - `tags:` — 내용에서 자동 추출
      - `source:` — 원본 raw 파일 경로
   c. wiki/ 적절한 하위 디렉토리에 파일을 생성한다.
   d. 기존 wiki 문서와 중복을 확인한다. 중복이면 기존 문서 갱신으로 전환한다.

#### merge 처리
   a. 클러스터 내 모든 노트를 읽는다.
   b. 핵심 사실을 추출하고 중복을 제거한다.
   c. `templates/`의 적절한 템플릿으로 통합 문서를 작성한다.
   d. frontmatter의 `source:`에 모든 원본 경로를 배열로 기록한다.
   e. 모순이 발견되면 `wiki/_meta/contradictions.md`에 기록한다.

#### selective 처리
   a. 각 노트의 중요도를 평가한다:
      - 핵심 사실/결정사항 포함 여부
      - 반복 절차/프로세스 포함 여부
      - 고유 정보(다른 노트에 없는 내용) 포함 여부
   b. 중요도가 높은 노트는 promote 또는 merge로 처리한다.
   c. 중요도가 낮은 노트는 manifest에 `보류`로 표시하고 비고에 사유를 기록한다.

11. 배치 처리 완료 시마다 `raw/.manifest.md`를 갱신한다:
    - 승격 완료 → `완료`
    - 일부 반영 → `부분` (비고에 사유)
    - 선별 제외 → `보류` (비고에 사유)
12. 배치 간 진행 상황을 출력한다:
    - `[배치 M/N] 처리: X건 promote, Y건 merge, Z건 보류`

### Phase 4: 색인 갱신 (1회)

13. 모든 배치 처리가 완료된 후 wiki/index.md를 갱신한다.
14. 관련 wiki/index/ 주제별 맵을 갱신한다.
15. 새로 생성된 문서 간 cross-link를 추가한다.

### Phase 5: 최종 보고

16. 최종 보고를 출력한다:

```
## Batch Ingest Report

### 처리 요약
| 항목 | 건수 |
|------|------|
| 총 대상 파일 | N |
| promote 처리 | X |
| merge 처리 | Y (Z개 클러스터) |
| selective 승격 | A |
| selective 보류 | B |
| 중복으로 기존 문서 갱신 | C |
| 오류/건너뜀 | D |

### 생성된 wiki 문서
| wiki 경로 | 모드 | source |
|-----------|------|--------|

### 클러스터 (merge)
| 클러스터 주제 | 통합 문서 | 원본 수 |
|---------------|-----------|---------|

### 보류된 파일 (selective)
| 파일 | 사유 |
|------|------|

### 발견된 모순
- ...

### Open Questions / TODO
- ...
```

## 절대 규칙

- raw/ 원본은 절대 수정하지 않는다.
- frontmatter 스키마(title, owner, status, updated, tags, source)를 준수한다.
- manifest 상태값은 `완료`, `미정`, `부분`, `보류`만 사용한다.
- wiki/index.md 갱신은 전체 배치 처리 완료 후 1회만 수행한다 (배치마다 하지 않는다).
- 근거 없는 내용을 쓰지 않는다. 불확실하면 `TODO` 또는 `Open question`으로 남긴다.
- 하나의 문서에 정책, 절차, 의사결정을 섞지 않는다.
- merge 시 원본의 핵심 사실이 누락되지 않도록 주의한다.
- 기존 wiki 문서와의 중복/모순을 반드시 확인한다.
- 배치 처리 중 오류가 발생하면 해당 파일을 건너뛰고 보고서에 기록한다. 전체 처리를 중단하지 않는다.

## 다른 스킬과의 관계

- **catalog**: Phase 1에서 미등록 파일을 자동으로 catalog 처리한다. 별도로 `/project:catalog`을 먼저 실행할 필요 없다.
- **ingest**: batch-ingest는 ingest의 핵심 로직(분석, 문서 생성, 중복/모순 확인)을 배치 단위로 수행한다. 개별 파일의 세밀한 처리가 필요하면 `/project:ingest`를 사용한다.
- **reindex**: Phase 4에서 자동으로 reindex를 수행한다. 별도 실행 불필요.
- **score**: batch-ingest 완료 후 `/project:score`로 신뢰도를 평가하는 것을 권장한다.
