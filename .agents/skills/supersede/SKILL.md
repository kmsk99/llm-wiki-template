# Supersede: 정보 대체 처리

기존 wiki 정보가 새 정보에 의해 대체될 때, 안전하게 갱신하고 이력을 남긴다.

## 입력

$ARGUMENTS

형식: `<새 소스 경로> [대체할 wiki 문서 경로]`

- 새 소스만 주어지면: 자동으로 대체 대상 wiki 문서를 탐색한다.
- 둘 다 주어지면: 지정된 wiki 문서를 갱신한다.

## 실행 절차

### Phase 1: 변경 범위 파악
1. 새 소스를 읽는다 (비텍스트면 Marker 파싱 우선).
2. 기존 wiki 문서 중 동일 주제를 다루는 문서를 찾는다.
3. **어떤 사실이 변경되었는지** 구체적으로 식별한다:
   - 수치 변경 (가격, 날짜, 수량 등)
   - 정책 변경 (규칙, 절차, 기준)
   - 상태 변경 (진행중→완료, 활성→폐기 등)
   - 구조 변경 (조직, 시스템 아키텍처 등)

### Phase 2: 대체 실행
4. 기존 문서에서 **변경된 부분만** 갱신한다 (전체 재작성 아님).
5. 변경된 사실은 값을 직접 갱신하고, 문서 하단 `## Change log` 섹션에 변경 이력을 기록한다:
   ```markdown
   ## Change log
   | 날짜 | 항목 | 이전 값 | 새 값 | 출처 |
   |------|------|---------|-------|------|
   | 2026-04-09 | 월 구독료 | 19,000원 | 29,000원 | raw/meetings/2026-04-09.md |
   ```
   - 이 방식은 Obsidian에서 가시적이고, 이전 값을 보존하면서도 본문을 깔끔하게 유지한다.
6. frontmatter를 갱신한다:
   - `updated:` — 오늘 날짜
   - `source:` — 새 소스를 추가 (기존 소스도 유지)

### Phase 3: 파급 효과 추적
7. 변경된 사실을 참조하는 **다른 wiki 문서**를 찾는다.
8. 해당 문서들도 갱신하거나, 갱신이 필요하다는 `TODO`를 남긴다.
9. output/ 문서 중 영향받는 것이 있으면 표시한다.

### Phase 4: 기록
10. 모순이 완전히 해소되었으면 `wiki/_meta/contradictions.md`에서 해당 항목을 제거한다.
11. 부분적으로만 해소되었으면 상태를 갱신한다.

## 출력

```
## Supersede Report

### 변경 사실
| 항목 | 이전 값 | 새 값 | 출처 |
|------|---------|-------|------|

### 갱신된 문서
- ...

### 파급 영향 (확인 필요)
- ...

### 해소된 모순
- ...
```

## 규칙
- 대체는 항상 근거(새 소스)가 있어야 한다. 추측으로 대체하지 않는다.
- 이전 값은 `## Change log` 테이블에 보존한다. 본문에는 최신 값만 유지한다.
- 파급 효과가 3개 문서 이상이면 사용자에게 확인을 요청한다.
- `## Change log` 섹션이 없는 문서에 처음 supersede할 때 섹션을 생성한다.
